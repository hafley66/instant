# Serializable reactive workflow runtime

## Objective

Define a portable, serializable execution model that composes:

- XState/SCXML-style statecharts
- FRP and reactive stream operators
- browser tab, DOM, and network event sources
- local Tauri effects and dashboard outputs
- JSON Logic predicates
- JSONata data extraction
- plugin-defined escape hatches

The runtime must support workflow changes without recompiling workflow definitions. A TypeScript reference implementation should be portable to Rust through a shared wire format and conformance fixtures.

## Semantic model

The canonical primitive is an event transducer:

```text
EventTransducer<Input, Output, InternalState>

input + internal state
  -> next internal state
  -> output events
  -> effects
```

An XState machine is a named stateful event transducer. An FRP operator is a composable stream transducer. `scan`, `combineLatest`, `withLatestFrom`, `mergeMap`, and `switchMap` are all instances of stateful or effectful transduction.

```text
external events
  -> stream operators
  -> machine transition
  -> context/state projection
  -> effects
  -> emitted events
  -> stream and machine graph
```

State is the latest projection of an event history. The portable model still exposes XState terminology for human and AI compatibility:

- `state.value`: active control location
- `context`: machine-local accumulated JSON data
- `event`: current input
- `guard`: pure transition predicate
- `action`: immediate effect
- `invoke`: long-lived effect or child actor
- `done`: child completion event
- `error`: effect failure event
- `self`: workflow instance identity and scope

Expression scope:

```json
{
  "event": {},
  "value": {},
  "self": {
    "id": "instance-1",
    "state": "checking",
    "context": {}
  },
  "env": {
    "tab": {},
    "page": {}
  }
}
```

## Existing semantic references

### AsyncAPI

[AsyncAPI 3.0](https://www.asyncapi.com/docs/reference/specification/v3.0.0) defines the event contract boundary:

- servers
- channels
- send/receive operations
- messages
- payload schemas
- headers
- correlation IDs
- protocol bindings
- request/reply descriptions

It does not define stream operators, statechart transitions, backpressure, cancellation, joins, retries, timers, or effect execution.

Use AsyncAPI for external event source and sink contracts.

### TeSSLa

[TeSSLa](https://www.tessla.io/documentation/) is the closest reference for a temporal stream specification language. It defines input streams, derived streams, temporal operators, trace-in/trace-out execution, a formal core, a compiler, and an interpreter. Its basic operators include `nil`, `unit`, `lift`, `last`, and `delay`.

Use TeSSLa as a reference for:

- stream equations
- temporal semantics
- formal operator definitions
- trace-based conformance testing
- a small stream-language core

### RTLola

[RTLola](https://arxiv.org/abs/1711.03829) extends stream specifications with real-time monitoring, asynchronous streams, windows, aggregates, triggers, static analysis, and generated monitor implementations. The framework has C and Rust implementation paths.

Use RTLola as a reference for:

- event-time versus processing-time semantics
- windowing
- trigger streams
- memory analysis
- deterministic evaluation order
- generated Rust monitors

### Siddhi

[Siddhi](https://siddhi.io/en/v5.1/docs/query-guide/) covers complex event processing:

- filters
- windows
- joins
- patterns
- sequences
- aggregations
- partitions
- event-time processing
- error streams
- snapshot and restore

Use Siddhi as a reference for event correlation, pattern matching, windows, and partitioned instances.

### Apache Beam

[Apache Beam](https://beam.apache.org/documentation/programming-guide/) defines keyed state, windows, watermarks, event-time timers, processing-time timers, triggers, late data, and replayable batch/stream processing.

Use Beam as a reference for:

- keyed instance state
- timers
- windows
- watermarks
- late events
- replay
- trigger semantics

### Apache Flink

[Flink](https://flink.apache.org/) and its [ProcessFunction](https://apache.googlesource.com/flink/%2B/master/docs/content/docs/dev/datastream/operators/process_function.md) expose the event/state/timer core of stateful stream processors.

Use Flink as a reference for event-time processing, keyed state, timers, checkpointing, and effectful event-driven applications.

### Kafka Streams

[Kafka Streams DSL](https://kafka.apache.org/43/streams/developer-guide/dsl-api/) distinguishes:

- `KStream`: events over time
- `KTable`: latest materialized projection
- `groupBy`: keyed instance partitioning
- `branch`: exclusive routing
- joins
- windows
- persistent state stores

Use this distinction for the relationship between event streams and synchronously readable latest values.

### SCXML and XState

[SCXML](https://www.w3.org/TR/scxml/) defines state values, hierarchical and parallel states, event queues, guards, entry/exit actions, data models, executable content, and run-to-completion semantics.

XState supplies a practical JSON/TypeScript-oriented statechart model. Use XState vocabulary in the public workflow format and SCXML semantics as the formal reference for machine transitions.

### CNCF Serverless Workflow

[CNCF Serverless Workflow](https://serverlessworkflow.io/blog/releases/release-100/) defines serializable event-driven workflows with tasks, events, retries, timeouts, parallel execution, subflows, correlation, and AsyncAPI integration.

Use it as a reference for durable task orchestration and effect lifecycle.

### Amazon States Language

[Amazon States Language](https://docs.aws.amazon.com/step-functions/latest/dg/concepts-amazon-states-language.html) provides a JSON state machine format with `Task`, `Choice`, `Parallel`, `Map`, `Pass`, `Wait`, `Succeed`, `Fail`, `Retry`, and `Catch` states. It also supports JSONPath and JSONata.

Use it as a reference for serializing workflow control, retry, timeout, branching, and parallel task behavior.

### Reactive Streams

[Reactive Streams](https://www.reactive-streams.org/) standardizes asynchronous stream processing and non-blocking backpressure. It intentionally leaves end-user DSLs and operator sets outside the specification.

Use it as a reference for backpressure vocabulary and asynchronous boundaries.

## Canonical runtime types

```text
Event<T>
  one occurrence with type, data, timestamp, scope, id, and causation

Stream<T>
  zero or more values over time

Signal<T>
  latest value plus future changes

Machine<S, C, E>
  statechart with state value S, context C, and event union E

Effect<T>
  one asynchronous operation with success, error, and cancellation

Scope
  global, plugin, tab, page, workflow, pipe, or event correlation scope
```

Canonical event:

```json
{
  "id": "evt_123",
  "type": "network.response",
  "time": 1784563200000,
  "scope": {
    "tabId": 12,
    "workflowId": "usage-monitor"
  },
  "data": {
    "status": 200,
    "body": {}
  },
  "causationId": "evt_122"
}
```

Canonical machine transition:

```text
transition(snapshot, event)
  -> snapshot
  -> effects
  -> emitted events
```

Reference TypeScript shape:

```ts
type TransitionResult = {
  snapshot: Snapshot
  effects: Effect[]
  events: Event[]
}
```

Reference Rust shape:

```rust
struct TransitionResult {
    snapshot: Snapshot,
    effects: Vec<Effect>,
    events: Vec<Event>,
}
```

## Operator semantics

Required first operator set:

```text
map
filter
scan
tap
route
merge
combineLatest
withLatestFrom
groupBy
mergeMap
switchMap
concatMap
window
delay
debounce
throttle
takeUntil
retry
timeout
```

XState/Rx correspondences:

```text
scan
  evolving state/context

route
  XState on-clause dispatch

combineLatest
  join current values from several signals

withLatestFrom
  sample another signal during an input event

mergeMap
  parallel invoked actors

switchMap
  replace and cancel the active invoked actor

concatMap
  queued child actors

takeUntil
  cancellation boundary

groupBy + mergeMap
  multiple keyed machine instances
```

`route` requires an explicit mode:

```text
exclusive
  one guarded branch wins by priority

broadcast
  all matching branches receive the event
```

Exclusive routing matches an XState machine's transition selection. Broadcast routing composes independent FRP consumers.

## Multiple state machine instances

Instances are keyed transducers:

```text
events
  -> keyBy(scope)
  -> groupBy(key)
  -> create/reuse machine per key
  -> merge outputs
```

Serialized instance policy:

```json
{
  "instance": {
    "key": "scope.tabId",
    "lifecycle": {
      "start": "firstMatchingEvent",
      "stop": "tab.closed",
      "idleTimeoutMs": 300000,
      "maxInstances": 100
    }
  }
}
```

Every instance has:

```json
{
  "self": {
    "id": "usage-monitor:tab:12",
    "machineId": "usage-monitor",
    "scope": {
      "tabId": 12
    },
    "state": "checking",
    "context": {}
  }
}
```

The runtime must define instance creation, reuse, eviction, cancellation, persistence, and rehydration.

## Expression languages

Use explicit tagged expressions:

```json
{
  "language": "json-logic",
  "value": {
    ">": [
      { "var": "self.context.usage" },
      0.8
    ]
  }
}
```

Core expression roles:

```text
literal
  JSON literal

path
  restricted field access

json-logic
  guards, filters, routing, retry predicates

jsonata
  JSON extraction and transformation

plugin
  registered application-specific expression
```

The first portable runtime should require JSON Logic for predicates. JSONata can remain an optional transform capability until TypeScript/Rust parity is established.

## Effect model

Effects are serializable descriptors interpreted by a capability registry:

```json
{
  "kind": "effect",
  "op": "dom.click",
  "input": {
    "selector": "#submit"
  },
  "timeoutMs": 3000,
  "retry": {
    "maxAttempts": 2,
    "backoffMs": 250
  }
}
```

Initial capability registry:

```text
dom.click
dom.fill
dom.waitFor
network.wait
tab.activate
self.context.assign
emit
tauri.command
plugin.call
```

Effect results:

```json
{
  "status": "ok",
  "value": {}
}
```

```json
{
  "status": "error",
  "error": {
    "kind": "timeout",
    "message": "dom.waitFor exceeded 3000ms"
  }
}
```

```json
{
  "status": "cancelled",
  "reason": "tab.navigated"
}
```

The pure transition evaluator must remain separate from effect execution. This makes the TypeScript and Rust transition cores testable against identical fixtures.

## Time and scheduling semantics

The specification must distinguish:

```text
event time
  when the observed event occurred

processing time
  when the runtime processed it

wall-clock time
  actual local clock used by timers

logical tick
  deterministic test scheduler step
```

Required rules:

- timestamps are explicit event data
- clocks are injected
- external events are ordered by the scheduler
- a transition runs to completion
- emitted events enter an explicit queue
- effects report completion through events
- cancellation disposes subscriptions and pending effects
- retries have explicit attempt and backoff semantics
- replay can use recorded event timestamps or logical ticks

## Impurity boundaries to stress-test

### Reentrancy

Effects can emit events while a transition is running. The runtime needs an internal queue:

```text
external event
  -> transition
  -> enqueue emitted events
  -> complete transition
  -> drain internal queue
  -> accept next external event
```

### Exclusive versus broadcast routing

XState transition selection is exclusive within a machine. FRP graphs may broadcast to several consumers. These must remain separate modes.

### Hot versus cold streams

```text
hot
  existing browser/event source shared by subscribers

cold
  new execution per subscriber
```

Network and tab sources are hot. Invoked workflows and replay pipes may be cold. The definition must state which behavior applies.

### Backpressure

Required policies:

```text
all
latest
first
drop
buffer
sample
```

### Cancellation

Navigation, tab closure, workflow reload, plugin unload, and explicit user cancellation must all be representable as events or lifecycle signals.

### State persistence

Persist:

```text
machine definition revision
state.value
context
instance scope
pending timer descriptors
causation position
```

Do not persist raw DOM nodes, Chrome ports, promises, AbortControllers, or stream subscriptions. Persist identifiers and reacquire runtime resources.

### Hot reload

Workflow JSON revisions can load without extension recompilation. Action/operator implementation changes require extension reload in MV3. Instance migration must be explicit:

```text
reload with compatible revision
  continue instance

reload with migration
  transform snapshot

reload with incompatible revision
  terminate or restart instance
```

## Recommended implementation order

1. Define the JSON event envelope and snapshot format.
2. Implement pure XState-style transition evaluation.
3. Add JSON Logic evaluation for guards and filters.
4. Add the event queue and run-to-completion scheduler.
5. Add `map`, `filter`, `scan`, `route`, and `tap`.
6. Add `combineLatest` and `withLatestFrom`.
7. Add keyed instance creation with `groupBy` semantics.
8. Add `mergeMap`, `switchMap`, and cancellation.
9. Add timers, retry, timeout, and backpressure policies.
10. Add effect descriptors and plugin capabilities.
11. Add persistence, replay, and revision migration.
12. Port the pure evaluator and conformance suite to Rust.

## Working design conclusion

The closest existing semantic stack is:

```text
AsyncAPI/CloudEvents
  event contract and envelope

TeSSLa/RTLola
  formal temporal stream equations

Siddhi
  event joins, windows, sequences, patterns, partitions

XState/SCXML
  named statechart control and context

Beam/Flink/Kafka Streams
  keyed state, timers, windows, replay, event-time semantics

Serverless Workflow/Amazon States Language
  serializable task control, retry, timeout, parallel, subflow

Reactive Streams
  asynchronous backpressure boundary
```

Instant should define a smaller canonical transducer IR informed by those systems. The TS and Rust implementations should share the IR, event semantics, effect descriptors, and conformance vectors. Human and AI authoring can use XState vocabulary and friendly JSON macros that compile into the same IR.

## Design decisions from follow-up

### State shape

The canonical snapshot has one state object:

```json
{
  "state": {
    "value": "checking",
    "usage": 0.86,
    "attempts": 1
  }
}
```

`state.value` is the reserved control-location field. The remaining state fields are accumulated instance data. A nested `context` field is not part of the core format.

The runtime supports all three state update forms:

```text
replace
  replace the complete state object

patch
  apply a JSON Patch operation list to the state object

set
  update one field by path
```

### Root subscription topology

The application has one runtime composition root, analogous to Redux store creation or the ReactDOM render root:

```text
root event source
  -> root Observable<Event>
  -> reducer/state subscriptions
  -> FRP combinator subscriptions
  -> effect subscriptions
  -> React/plugin/dashboard subscriptions
```

The root owns the scheduler, event ingress, subscription disposal, persistence hooks, and runtime capabilities. React rendering is a subscriber/projection. Plugins attach subscriptions to the root or to a declared child scope. Reducers are pure subscription targets. Effects are explicit Observable branches.

There is no middleware layer. Event transformations and effect execution are visible as Observable composition.

### Instance routing

Instance partitioning is URL-like routing. A workflow definition declares a key expression and lifecycle event patterns:

```json
{
  "instance": {
    "key": "event.partitionKey",
    "startOn": "page.matched",
    "stopOn": ["page.closed", "workflow.cancelled"]
  }
}
```

The core does not name tabs. A browser adapter may produce `tab:12`, `page:abc`, or another partition key. Other adapters may use a Slack channel, file path, WebSocket connection, request ID, or workflow invocation ID.

### FRP surface

Start with named combinator nodes. A linear pipe syntax is deferred.

```json
{
  "kind": "mergeMap",
  "source": "network.responses",
  "project": {
    "flow": "extractUsage"
  }
}
```

Initial combinators:

```text
merge
mergeMap
switchMap
concatMap
combineLatest
withLatestFrom
groupBy
scan
filter
map
takeUntil
```

`mergeMap` expresses concurrent child flow instances. `switchMap` expresses replacement and cancellation of the previous child instance. `groupBy` plus `mergeMap` expresses one machine per routing key.

### Observable lifecycle

Use the common Observable notification model:

```text
next(value)
error(error)
complete()
```

Effects are monadic Observable-producing computations. They can emit events, state-update events, lifecycle events, or ordinary values through `next`. They terminate through `error` or `complete`.

```text
Effect<A> = Observable<A>
```

The runtime may encode effect commands, state updates, and lifecycle changes as typed events. This keeps effects and state changes inside the same event algebra while preserving Observable completion and error behavior.

### Definition changes

Workflow definitions are refreshed and restarted. Definition migration is outside the initial scope.
