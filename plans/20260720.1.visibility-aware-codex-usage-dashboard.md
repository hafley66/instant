# Visibility-aware Codex usage flows and comparison dashboard

## Objective

Use JSON-Rx named flows and Rx operators to:

1. Poll Metrics only while its panel is visible.
2. Consume Codex rate-limit snapshots and update notifications.
3. Expose a shared latest `codex.usage` flow.
4. Render `claude.usage` and `codex.usage` side by side through generic Metrics
   components.

The implementation follows
[the JSON-Rx specification synthesis](../books/json-rx-automation/4_json-rx-specification-synthesis.md).
It does not introduce a first-class query abstraction. Visibility, refresh,
freshness, cancellation, recurrence, and sharing are composed through ordinary
sources and operators.

## Current reality

Metrics currently runs:

```text
timer(0, 5000)
  -> exhaustMap(activity_rule_matches)
  -> loaded | failed events
  -> scan Metrics state
```

The subscription starts when `MetricsDashboardPanel` mounts. Dockview can hide
panels without making React mount lifetime a reliable visibility signal.

The dashboard envelope already supports independent streams:

```ts
type DashboardEmission = {
  ruleId: string;
  url: string;
  ts: number;
  matches: Record<string, unknown>[];
  stream: string;
  schema: JsonSchema;
};
```

The installed Codex app-server protocol exposes:

- `account/rateLimits/read`
- `account/rateLimits/updated`
- `account/usage/read`

The first implementation needs rate-limit snapshots and sparse updates.

## Type signatures

### Host visibility source

```ts
type PanelVisibility = {
  visible: boolean;
  documentVisible: boolean;
  intersecting: boolean;
};

function panelVisibility$(element: HTMLElement): Observable<PanelVisibility>;
```

The host's external dispatch union gains members for panel visibility and
Codex app-server notifications. Dispatch selects a source by its discriminant,
validates the event, and pushes it into that source instance. Named flows refer
to the source; dispatch does not contain flow or dashboard routing.

### Codex host operations

```ts
type CodexHost = {

  rateLimitsUpdated$: Observable<CodexRateLimitsUpdate>;

type CodexUsage = {
  primary_percent: number | null;
  primary_resets_at: string | null;
  secondary_percent: number | null;
  secondary_resets_at: string | null;
  credit_balance: number | null;
  has_credits: boolean | null;

```

### Flow catalog

```ts
type FlowCatalog = {
  flow: <T extends JsonValue>(ref: string) => Observable<T>;
};
```

The serialized document names `metrics.matches`, `codex.usage.whileVisible`,
and `codex.usage`. The TypeScript runtime resolves those names into
Observables. Instant chooses which flows to subscribe to.

## Flow definitions

### Visibility-aware Metrics reads

Reference RxJS:

```ts
const metricsMatches$ = panelVisibility$.pipe(
  map((value) => value.visible),
  distinctUntilChanged(),
  switchMap((visible) => {
    if (!visible) return NEVER;

    return merge(
      of("immediate"),
      timer(5_000, 5_000),
    ).pipe(
      exhaustMap(() => loadMetricMatches()),
    );
  }),
  shareReplay({ bufferSize: 1, refCount: true }),
);
```

Temporal topology:

```text
document visibility ─┐
panel intersection ──┴─ combine -> distinct visible boolean
                                    │
                                    └─ switchMap
                                         ├─ hidden: NEVER
                                         └─ visible:
                                              immediate ─┐
                                              interval ──┴─ merge
                                                            │
                                                            └─ exhaustMap [DB]
                                                                  │
                                                                  └─ shareReplay(1)
```

The switch contains the interval and native read. Hiding the panel disposes
the complete visible routine, including a cancellable read. `exhaustMap`
prevents overlapping reads while visible. `shareReplay(1)` remains outside the
switch so subscribers can receive the latest result across visibility changes.

### Codex usage

Snapshot and update events need one recurrent value. The reducer replaces on a
complete snapshot and merges on a sparse update.

```ts
type CodexSnapshotEvent =
  | { type: "snapshot"; value: CodexRateLimitsSnapshot }
  | { type: "update"; value: CodexRateLimitsUpdate };

const codexUsageWhileVisible$ = merge(
  merge(
    of("immediate"),
    timer(60_000, 60_000),
  ).pipe(
    exhaustMap(() => codexHost.rateLimitsRead()),
    map((value) => ({ type: "snapshot" as const, value })),
  ),
  codexHost.rateLimitsUpdated$.pipe(
    map((value) => ({ type: "update" as const, value })),
  ),
).pipe(
  scan(applyCodexSnapshotEvent, emptyCodexSnapshot),
  map(normalizeCodexUsage),
);

const codexUsage$ = panelVisibility$.pipe(
  map((value) => value.visible),
  distinctUntilChanged(),
  switchMap((visible) => visible ? codexUsageWhileVisible$ : NEVER),
  shareReplay({ bufferSize: 1, refCount: true }),
);
```

The first visible subscriber starts snapshot reconciliation and app-server
updates. Hiding the panel releases both. Returning visible creates a new inner
instance and obtains an immediate complete snapshot.

## Observable lifecycle diagnostics

Each named flow and selected operator boundary publishes a separate diagnostic
trace:

```text
subscribe
next*
error | complete | unsubscribe
finalize
```

Operator-specific trace outcomes include:

- visibility switched to `NEVER`
- `switchMap` cancelled the prior visible routine
- `exhaustMap` ignored a refresh tick
- snapshot read subscribed, returned, failed, or finalized
- sparse update merged into the current snapshot
- `shareReplay` acquired or released its upstream subscription

Diagnostics observe the flow. They do not serve as dashboard or persistence
sinks.

## Instance timelines and lifetimes

### Metrics panel

```text
panel mounts
  -> create visibility Observable
  -> subscribe metrics.matches
  -> visible branch loads immediately

panel hidden
  -> switchMap unsubscribes visible branch
  -> timer and active work finalize
  -> shared latest value remains available in the outer flow instance

panel visible
  -> switchMap creates a new visible branch
  -> load immediately

panel unmounts
  -> final subscriber leaves
  -> refCount releases upstream visibility source
```

### Codex host

```text
first active codex.usage subscription
  -> acquire app-server lease
  -> read complete rate-limit snapshot
  -> subscribe sparse update notifications

snapshot or update
  -> scan produces next complete snapshot
  -> normalization produces next CodexUsage

visible routine finalizes
  -> release update subscription
  -> cancel pending read where host supports cancellation
  -> release app-server lease

last host lease released
  -> host may terminate app-server after an idle timeout
```

The app-server process has host-managed lifetime. A JSON-Rx subscription owns
a lease, not the process directly.

## Storage, reads, writes, and uniqueness

### Shared latest values

`shareReplay({ bufferSize: 1, refCount: true })` supplies one latest value per
running named-flow instance. There is no additional query cache or retain
abstraction.

The replay buffer is in memory and disappears when the runtime instance is
discarded according to RxJS reset semantics. Persistent history remains an
Instant host concern.

### Dashboard persistence

Codex observations enter the existing activity database as dashboard
emissions:

```text
stream = codex.usage
ruleId = codex-rate-limits
url = codex-app-server://account/rateLimits
```

Write sequence:

```text
complete normalized CodexUsage
  -> JSON Schema validation
  -> dashboard envelope
  -> localhost/native ingest
  -> append rulematch activity row
```

Sparse updates are merged by `scan` before persistence. Every stored dashboard
record is a complete normalized observation. Raw app-server responses and
authentication material are excluded.

Read sequence:

```text
visible Metrics panel
  -> metrics.matches flow
  -> one native activity_rule_matches read
  -> derive rows for both selected streams
```

The outer Metrics panel performs one database read. Side-by-side stream views
do not create independent reads.

### Metrics UI state

Persist through `pluginState`:

```ts
type MetricsUi = {
  layout: number[];
  comparisonStreams: string[];
  comparisonLayout: number[];
};
```

The selected stream names must be distinct. The initial UI permits one or two
streams. Existing one-stream behavior remains the default.

## Side-by-side dashboard composition

```tsx
<MetricsDashboardPanel>
  <MetricsComparison streams={[left, right]}>
    <MetricStreamView stream={left} rows={rows} />
    <MetricStreamView stream={right} rows={rows} />
  </MetricsComparison>
</MetricsDashboardPanel>
```

Implementation boundaries:

| Concern | Component or module |
| --- | --- |
| visibility source | browser host adapter |
| named flow compilation | `src/lib/json-rx/` |
| Metrics database flow | Metrics runtime |
| per-stream derivation | pure Metrics functions |
| horizontal comparison | `react-resizable-panels` |
| chart | Vega-Lite through `vega-embed` |
| history | `TreeTable` |
| persisted UI selection | `pluginState` |

Each `MetricStreamView` owns cards, chart, and history for one stream. The
horizontal comparison split and each vertical chart/history split persist
independently.

## Codex normalization

Prefer `rateLimitsByLimitId.codex` when present. Fall back to the backward-
compatible `rateLimits` snapshot. Convert Unix reset timestamps to ISO 8601.

```text
GetAccountRateLimitsResponse
  -> choose Codex bucket
  -> replace or merge snapshot through scan
  -> map primary and secondary windows, credits, and plan
  -> validate CodexUsage
  -> next codex.usage
```

The output JSON Schema labels percentage fields and declares `minimum: 0` and
`maximum: 100`. Generic Metrics rendering can then create cards and chart
series without a Codex-specific dashboard component.

## Host adapter decision

The Codex source needs one explicit host implementation:

1. Sprefa adapter using the streaming shell/process host contract.
2. Direct Instant adapter speaking the Codex app-server protocol.

The flow definitions, reducer, normalization, dashboard envelope, and UI remain
the same. Process spawning and credentials do not enter the portable JSON-Rx
package.

## Backpressure and scheduler boundary

The TypeScript reference uses RxJS push semantics. The Rust implementation uses
`Stream` and `StreamExt` with `poll_next` readiness. This plan requires no
high-volume unbounded source, but conformance must still record:

- queue capacity at the app-server transport
- overflow behavior for update notifications
- cancellation behavior for snapshot reads
- ordering between snapshots and sparse updates
- scheduler or executor used by timers and protocol I/O

A scheduler controls when work runs. Demand and bounded queues remain separate
host-profile concerns.

## Implementation stages

### 0. Flow conformance fixtures

- Add virtual-time fixtures for visible, hidden, re-visible, interval, read,
  cancellation, sharing, and replay timelines.
- Assert values, subscriptions, finalization, and ignored refresh ticks.

### 1. Source and named-flow compilation

- Define source and flow JSON shapes in `0_types.ts`.
- Compile refs and ordered pipes into RxJS Observables.
- Preserve direct RxJS operator names and semantics.

### 2. Browser visibility source

- Combine document visibility and panel intersection.
- Emit distinct visibility values.
- Dispose observers and listeners when the source unsubscribes.

### 3. Metrics flow migration

- Replace unconditional polling with the visibility-switched flow.
- Keep one read active through `exhaustMap`.
- Share and replay the latest result.

### 4. Generic comparison UI

- Extract `MetricStreamView`.
- Add two stream selectors and a horizontal resizable split.
- Persist stream selection and layouts.

### 5. Codex host source

- Implement the selected host adapter.
- Read complete snapshots and consume sparse updates.
- Normalize and ingest `codex.usage`.
- Release resources when the visible inner flow finalizes.

### 6. Verification

- Vitest virtual-time flow conformance tests.
- Sanitized Codex protocol fixtures.
- Fake JSON-RPC app-server integration test.
- Metrics Playwright test with Claude and Codex side by side.
- Golden screenshot after both Vega views report render-ready.
- `just check`, `just build`, `just cargo-check`, `just ext-build`, and full
  Vitest.

## Completion criteria

- Hidden Metrics panels produce no periodic database reads.
- Returning visible starts one immediate read and one interval timeline.
- Refresh bursts do not overlap reads.
- `shareReplay(1)` prevents duplicate upstream work for multiple subscribers.
- Codex snapshots and sparse updates produce complete normalized values.
- Claude and Codex render simultaneously through generic Metrics components.
- Hiding or closing the dashboard releases timers and source leases.
- Diagnostics identify subscription, cancellation, ignored, error, and
  finalization events.
- No authentication token, cookie, or raw Codex response is persisted.
