# Production JSON-Rx v2 Codex and Metrics comparison

## Type signatures

```ts
type DashboardEmission = {
  ruleId: string;
  url: string;
  ts: number;
  matches: Record<string, unknown>[];
  stream: string;
  schema: JsonSchema;
};

type AutomationV2Runtime = {
  automation: AutomationV2;
  roots: Record<string, Observable<DashboardEmission>>;
  canonicalIr: string;
};

function compileAutomationV2(
  input: unknown,
  sources: Record<string, Observable<RuntimeSource>>,
): AutomationV2Runtime;

type CodexHostAdapter = {
  rateLimitsRead: () => Observable<CodexRateLimitsSnapshot>;
  rateLimitsUpdated$: Observable<CodexRateLimitsUpdate>;
};

function codexUsageRuntime(host: CodexHostAdapter): AutomationV2Runtime;
```

The v2 compiler accepts validated source bindings and compiles source, merge,
machine, project, and shareReplay nodes into RxJS Observables. The Codex
definition maps the two host operations into a merged scan-backed machine and
projects a complete nullable `CodexUsage` record. The Metrics panel consumes
one `activity_rule_matches` read and derives one or two selected stream views.

## Instance lifetimes

The compiled flow definition is created once per panel/runtime owner. A shared
flow has one active source subscription and one scan accumulator for all
overlapping subscribers. The final subscriber releases the upstream source via
`refCount`; a later subscriber creates a fresh accumulator. A Codex adapter
lease is represented by the Observable subscription and is released when that
subscription ends. The fake adapter follows the same subscription behavior.

## Storage, read/write sequence

Codex host values are converted into normalized host events before entering the
v2 machine. `account/rateLimits/read` replaces the machine context with a full
snapshot. `account/rateLimits/updated` patches only supplied sparse fields over
the nullable initial context. Each machine output becomes one
`DashboardEmission`, then the existing host ingest path persists it as a
`rulematch` row. Metrics polls `activity_rule_matches(limit=500)` once per
interval and derives both selected streams from that returned array. UI layout
and stream selection persist through the `metrics` pluginState slice.

## Uniqueness assumptions

Stream names are stable dataset identities and are unique within the compiled
automation output set. V2 node IDs are unique across the circuit. One
`rulematch` row represents one emission envelope, while `matches` may contain
multiple records. Metrics comparison keys combine stream, timestamp, rule, and
row index so duplicate timestamps remain renderable. Codex normalization emits
all nullable schema fields on every observation, including update-before-snapshot
events.

## Scope boundary

Repository inspection found no live Codex app-server event/read adapter. This
change adds the typed adapter interface and deterministic fake adapter coverage.
Process spawning, credentials, protocol transport, and Sprefa event forwarding
remain pending host work.
