# Session: json-rx-next-steps

## Goal
Continue JSON-Rx as a small portable specification for process-local reactive circuits, using RxJS 7.8 as the first semantic profile and Instant as the first practical consumer.

## Problem/Context
The design has enough evidence to proceed, but expanding operator count, protocol profiles, durability, statecharts, or Instant integration simultaneously would hide semantic mistakes. The adversarial reviews identified missing normative AST, notification, identity, ordering, cancellation, sharing, queue, and conformance rules.

## Solution/Approach
Use a narrow vertical sequence:
1. Freeze the current lab and terminology.
2. Write one versioned JSON Schema and canonical IR for the five proven nodes.
3. Add runtime JSON validation with an off-the-shelf validator.
4. Define virtual-time fixtures for same-tick ordering, reentrancy, cancellation, and sharing reset.
5. Add operators one at a time only when an Instant use case requires them.
6. Build one real Instant automation in an isolated adapter after the fixture suite is authoritative.
7. Attempt Rust lowering only after TypeScript semantics and fixtures stop changing.
8. Keep Kafka, Temporal, SCXML, and relational extensions as non-normative research until required.

## Key Insights
- URI templates identify definitions; canonical filled URLs identify instances.
- One graph object and one shared upstream acquisition are separate claims.
- Tests exposed real semantic bugs before integration.
- Operator-owned queues make backpressure part of operator semantics as well as host/channel profiles.
- CSP, Rx, and Datalog occupy communication, sequence, and relational-fixpoint layers respectively.
- The first portable profile should remain an exact, constrained RxJS subset.
