## Solution/Approach
- **Zoom (3):** state.ts `zoom:number` (persisted, default 1). main.ts imports `getCurrentWebview` from `@tauri-apps/api/webview`; `applyZoom/nudgeZoom/resetZoom` (clamp 0.5–2.0, step 0.1); keymap cmds `app.zoomIn` (`$mod+Equal`/`$mod+Shift+Equal`), `app.zoomOut` (`$mod+Minus`), `app.zoomReset` (`$mod+Digit0`); `applyZoom()` in boot before registerBuiltin. **Required `core:webview:allow-set-webview-zoom` in src-tauri/capabilities/default.json** — without it setZoom throws and the `.catch(console.error)` swallows it silently (this was the user's "no zoom, how zoom?" bug).
- **Toolbars (2):** tablepanels.tsx `TmuxToolbar` (sort select bound to sessionSort, +claude/+opencode quick-launch, new-shell input) + `WtToolbar` (scan-root input + Scan, ☆/★ Focus toggle, count). New `TmuxBridge`/`WtBridge` fields wired in registerV2Bridges. Panels became `.v2-panel` flex columns (`.wt-scan` toolbar above flex-fill `.panel-scroll`). Focus on-state CSS broadened from `#wt-focus.on` to also match `.wt-focus-btn.on`.
- **TreeTable infra:** treetable.tsx gained `virtual?` (react-virtual spacer-`<tr>` pattern mirroring table.ts virtualTable, own `.tt-scroll` container), `getRowCanExpand?(row)` (twisty before children load), `onToggleExpand?(row,willExpand)` (fires before toggle; lazy loaders hook here). Virtualizer created unconditionally (hooks rule), consulted only when `virtual`.
- **Files tree (4):** FilesPanelV2 + setFilesPanel bridge. Lazy: root listing in store.files, per-folder children in new store field `fsChildren: Record<string,FsEntry[]>` loaded on expand via list_dir keyed by path. `loadFsRoot` (resets children), `loadFsChildren`, `fsRows`/`toFsRow` build display-ready FsRow (glyph/date/type/size formatted in main.ts). Folder dbl-click = new root, file click = preview, file dbl-click = paste. Legacy `#fs-list` id kept on the scroll wrapper so the file ctx-menu + drag-entity hooks still match. Removed renderFilesPanel + browseTo + wireChrome fs wiring + its store.subscribe/boot call; ctx "Up one folder" → filesGoUp.
- **Worktrees tree-data (6):** WtTreeRow (org→clone→leaf) built by main.ts `wtTreeRows` from `buildTree(focusRows())`; one TreeTable with `getSubRows`+`defaultExpandedAll` (name column = tree/chevron). Removed flat WtRow/WT_COLUMNS/wtRows/wtGestures. Added `.dtable-row:hover .wt-actions/.wt-star` reveal (old reveal keyed on `.wt-node:hover`).

## Key Insights
- setZoom (and other tauri webview/window calls) silently no-op without the matching capability; the `.catch` hides it. Capabilities are baked into the Rust binary → require `tauri dev` restart, not just cmd+R.
- Component panels (PanelDef.component) use useApp() → re-render on ANY store change (useSyncExternalStore over whole state), so no per-panel store.subscribe is needed; deleting the vanilla render fn + its subscription is the conversion.
- TreeTable virtualization uses spacer-`<tr>`s (before/after height) inside the same tbody (mirrors table.ts) so column widths stay aligned — no absolute positioning.
- Lazy tree pattern: getRowCanExpand=is_dir shows the twisty before load; onToggleExpand triggers list_dir into a store map; getSubRows reads the map; immutable `{...prev,[path]:entries}` bumps the store ref → re-render. Expanded state is TreeTable-internal, keyed by getRowId (path), survives data changes.
- Mutable module vars (wtAddingClone) can't drive React re-renders → move to the store. The dead vanilla renderTree still referenced it, so repointed those 3 dead refs to store.get/set rather than excising renderTree (kept that as a separate task).
- Reuse legacy ids (`#fs-list`, `#activity-table`) on the new React scroll wrappers so the global `closest("#id tr.dtable-row")` ctx-menu + `[data-entity-kind]` drag handlers keep matching with zero changes.
