# Session: json-rx-instant-sprefa-mvp

## Goal
Define a portable, serializable JSON-Rx specification for process-local reactive circuits that follows RxJS semantics, lowers to TypeScript/Rust hosts, supports channels and backpressure explicitly, and provides the programmable backbone for Instant browser automations and dashboards.

## Current State
- Instant's Claude usage network capture and Metrics dashboard were made functional earlier in the session.
- JSON-Rx exploration has been consolidated into a large specification synthesis and supporting book chapters.
- Two independent gpt-5.6-terra adversarial reviews completed and identified normative gaps.
- Runtime identity is now modeled as URI templates for reusable definitions and canonical filled URLs for instances.
- An isolated MVP exists under `labs/json-rx-mvp/`; it does not import from or modify running Instant code.

## Problem/Context
The original Instant requirement was configurable local automation without recompilation: browser/tab matching, DOM and network capture, timers, conditional actions, structured extraction, local effects, diagnostics, and generic dashboard emissions. Linear Rule JSON was insufficient for temporal composition, cancellation, recurrence, concurrency, sharing, and protocol streams. RxJS supplies the desired operator vocabulary. Rust hosts should use `Stream`/`StreamExt` rather than an all-push Rx port. Backpressure must account for source/channel boundaries and operator-owned queues.

The user wants the spec to remain recognizably RxJS, use flat normalized type/definition maps, support point-free named references, permit code generation/scaffolding and conformance checking, and avoid invented abstractions where established semantics exist.

## Solution/Approach
### Layering
```text
JSON Schema
  values, errors, parameters, document schemas

URI identity
  URI template = reusable definition
  canonical filled URL = runtime instance

Rx algebra
  Observable sources, operators, flattening, sharing, lifecycle

CSP/channel profile
  send/receive direction, capacity, rendezvous, ordering, closure

Host execution profile
  RxJS push, Rust Stream pull, MPSC, Go channels

Optional extension profiles
  Kafka-like logs, Temporal durability, SCXML, Beam, differential dataflow
```

### Protocol modeling
- HTTP: cold operation producing one or many response notifications.
- SSE: long-lived receive source; related sends use a separate HTTP operation.
- WebSocket: shared host resource exposing incoming Observable source plus send operation.
- Duplex protocols use two unidirectional capabilities: receive Observable and send operation/Observer.

### URI identity
```text
definition: jsonrx://instant/flows/codex/usage/{accountId}
instance:   jsonrx://instant/flows/codex/usage/account-123?interval=60000
```
Canonicalization sorts query keys, validates typed parameters, fills defaults consistently, normalizes encoding, and rejects unknown/missing parameters. Multiple references to one canonical URL resolve to one graph object. `share`/`shareReplay` still determine upstream acquisition behavior.

### Isolated MVP
Supported AST nodes:
```text
source
of
map.get
switchMap.ref
shareReplay(1, refCount=true)
```
Proved:
- JSON serialization round trip.
- URI-template expansion and canonical instance equality.
- One Observable object per canonical flow URL.
- One shared source acquisition for overlapping subscribers.
- Outer-value projection into inner flow path/query parameters.
- `switchMap` release of the prior inner instance.
- Serializable `next | error | complete` notifications.
- Stable node IDs and ordered lifecycle traces.

The first test run caught two real semantic bugs:
1. Query defaults overwrote explicit query values already present in an instance URL.
2. The registry cached an Observable wrapper while reconstructing `shareReplay` inside each subscription, causing duplicate source acquisitions.
Both were fixed in the lab.

## Adversarial Review Results
Both Terra reviewers converged on these blockers before portability or proof claims become normative:
1. Define a versioned JSON AST with parameter binding, lexical scope, reference rules, stable node IDs, and cycle/guarded-recursion rules.
2. Define `Notification<T,E> = next(T) | error(E) | complete` separately from domain `Event<T>`.
3. Define runtime graph, source, subscription, operator, and inner-instance identity.
4. Define same-time ordering, reentrancy, scheduler microsteps, timer laws, and merge fairness.
5. Replace provisional derived-operator claims with total lowerings or retain required lifecycle primitives in the kernel.
6. Put admission, ordering, capacity, and overflow semantics on operator-owned queues such as `concatMap` and bounded `mergeMap`.
7. Define host-operation cancellation as abort-confirmed, cooperative, detach-and-suppress, or unsupported.
8. Pin the initial semantic profile to RxJS 7.8.x.
9. Treat current function-valued `src/lib/json-rx` definitions as host implementation bindings rather than serializable AST.
10. Expand conformance fixtures to include observer-specific notifications, source acquisition, same-time order, cancellation acknowledgments, queue admission, and diagnostics.

Instant-specific review findings:
- Codex snapshot/update reconciliation needs monotonic revisions/cursors to prevent stale snapshot regression.
- Catalog lookup can reconstruct separate shared graphs unless canonical instance memoization is explicit.
- Visibility needs a truth table, initial emission, panel-active input, and visibility epochs.
- Dashboard persistence needs event identity, producer/source, observed timestamp preservation, schema version/fingerprint, and duplicate behavior.

## Sprefa Connection
Read `/Users/chrishafley/projects/sprefa/README.md`. Sprefa already implements a relational reactive sibling model:
```text
source facts -> least fixpoint per tick -> sink/demand rows -> effects -> later tick
```
Mappings:
- rule body/head: relational dataflow.
- recursion: fixpoint/feedback.
- `@next`: one-tick delay/register.
- `clock`/`every`: temporal value and periodic trigger.
- `@async`: unbounded mergeMap-like effect fanout.
- `@stream`: Observable-producing host operation.
- demand relations: effect request streams.
- `effect_log`: materialized operation lifecycle.
- `(repo,path,rev)`: stable coordinate compatible with URI instance identity.

Sprefa uses sets and stabilized relation extensions. Rx uses ordered notification sequences with multiplicity. Bridging requires relation-valued Observables or relation deltas. Encoding Rx into relations requires occurrence identity and sequence/logical-time columns.

## Key Insights
- URI templates eliminate abstract runtime “scope” terminology: templates identify definitions and canonical filled URLs identify instances.
- Caching an Observable object and sharing its upstream subscription are separate semantics.
- Backpressure is not only a scheduler or source concern. Flattening and sharing operators own admission queues too.
- CSP models communication topology; Rx models temporal sequence transformation; Datalog models relation transformation and fixpoint convergence.
- Two unidirectional capabilities model duplex protocols: incoming Observable plus outgoing operation.
- Existing-code verification can prove structure and run conformance fixtures; arbitrary temporal behavior cannot be statically proven from an exported type alone.
- The smallest credible portable profile is an exact RxJS version plus a constrained AST and shared virtual-time fixtures.

- Which queue/admission fields belong in portable operators versus host profiles?
