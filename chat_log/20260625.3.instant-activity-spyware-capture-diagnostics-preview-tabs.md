## Problem/Context
- Screenshots weren't firing on click. capture::take (src-tauri/src/capture.rs) silently no-ops on 6 conditions: Record off, instant window focused (clicking inside the app never shoots, by design), frontmost app excluded, <350ms throttle, Screen Recording denied (screencapture -x silently fails), or CGEventTap not installed (Accessibility/Input Monitoring). Zero feedback => "nothing happens". Most likely cause on `tauri dev`: binary signature rotates each rebuild => macOS drops Screen Recording grant.
- copy/paste (cmd+C=8, cmd+V=9 in lib.rs:175) go through the SAME capture path, so they work exactly when os shots work.
- files source: only logFileOpen (main.ts:1668) on DOUBLE-CLICK in the Files panel; no OS-wide fs watcher. session source: logTabVisit (main.ts:127) on tab focus. browser: extension -> 127.0.0.1:8787/ingest.
- Activity preview was an inline pane; user wants any file-backed row (screenshot PNG or logged file) to open the shared file preview (renderPathInto: images via read_image, code via shiki) in a split-RIGHT dock tab.
- next/prev tab + cmd+1..9 only walked api.activeGroup.panels => stayed in one pane.

## Solution/Approach
- Capture observability: capture::take now emits a `capture-status` event {kind,ok,reason,ts} at every outcome (shot saved or precise skip reason). New TapActive(Arc<AtomicBool>) flag set true when the CGEventTap runloop starts (lib.rs spawn_input_taps), false if creation failed. New commands capture_permissions (CGPreflightScreenCaptureAccess + AXIsProcessTrusted + tap_active) and capture_request_screen (CGRequestScreenCaptureAccess) via extern "C" #[link(name="ApplicationServices", kind="framework")]. Front: ActStatusBar in tablepanels.tsx shows permission banners (Screen Recording [Grant] button, input-access-off warning) + last-gesture line (`✓ shot: copy` verifies copy/paste, or `skipped: click — <reason>`). Store gained capturePerms/captureStatus (runtime); main.ts refreshCapturePerms() + listen("capture-status").
- Preview-as-tab: addPreviewPanel(reactdock.tsx) takes direction "within"|"right"; first preview splits right of terminals, subsequent stack within the existing preview group. openPreviewPanel threads direction. ActRow gained source + filePath (=shot || (source==="files"? text)); onRowClick opens openPreview(filePath) -> openPreviewPanel(path, undefined, "right"). Removed inline ActPreview + loadShot.
- Cross-pane nav: new allPanelIds() in reactdock (iterate api.groups -> all panel ids, like termPanelOrder). visualTabIds() in main.ts now uses allPanelIds() instead of groupPanelIds(); focusPanelById uses setActive() which also activates the target group.
- Security (committed ab220be): sanitizePaste() strips C0+DEL before write_pty (kills activity-row newline->auto-run injection); create_workspace branch.replace(['/','\\'],"-") path-traversal fix; renderConfigPanel escapeHtml; .v2-panel sets panel bg/fg (activity rows were dark-on-dark because activity uses .fs-list not .panel-scroll).

## Key Insights
- oddbird css-anchor-positioning polyfill builds its anchor registry from STYLESHEETS, not inline styles; anchor-name/anchor() must be emitted into a real <style> block. Use explicit two-arg anchor(--name side); nameless anchor()+position-anchor failed. Gate on CSS.supports("left: anchor(--x right)") (the function), not anchor-name (the property).
- Tauri capabilities AND extern framework symbols only error at LINK time; cargo check passes, cargo build (the instant(bin) step) is what proves linking. macOS Screen Recording/Accessibility are per-binary; `tauri dev` rebuilds drop the grant.
- useApp() re-renders v2 panels on ANY store change, so capture state can live in the store with no explicit subscribe.
- Dockview: panel.api.setActive() activates the panel AND its group; api.groups is the established visual-order source (termPanelOrder relies on it).
